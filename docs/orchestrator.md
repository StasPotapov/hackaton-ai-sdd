# Формализация: самоэволюционирующийся оркестратор SDD

> Этот документ формализует **расширенное видение**: от «скилла-носителя процесса,
> доводящего задачу до готовности» (база `Ready`, см. [vision.md](vision.md)) — к
> **оркестратору полного SDD-конвейера**, который ведёт задачу от сигнала до PR.
> Он не отменяет базу `Ready`, а надстраивается над ней: контур «Применяет» (готовность)
> становится одним из этапов более длинного конвейера.

---

## Суть в одном предложении

**Настраиваемый под любую команду оркестратор SDD-процесса: ловит сигнал из любого канала,
проводит его через заданный командой конвейер (интервью → ТЗ → аналитика → дизайн → код) до
PR — и сам эволюционирует, подсказывая, как улучшить этот процесс.**

Отличие от базы `Ready`: там главный артефакт — **готовое ТЗ**, здесь главный артефакт —
**код** (доведение до PR), а сам процесс не захардкожен, а собирается в конструкторе.

---

## Что это (три опоры)

| Опора | Роль | Ключевой вопрос, на который отвечает |
|---|---|---|
| **1. Оркестратор** | Исполняет процесс | *«Где сейчас задача и что с ней делать дальше?»* |
| **2. Конструктор процесса** | Задаёт процесс | *«Как выглядит SDD именно у этой команды?»* |
| **3. Самоэволюция** | Улучшает процесс | *«Как сделать этот процесс лучше на следующей итерации?»* |

Целевая аудитория: команды, у которых **основной артефакт — код**. Одиночный режим
(«запуск на одного») поддерживается как частный случай.

---

## Смещение акцента относительно базы `Ready`

Формализация сознательно двигает три вещи — это важно держать явно, чтобы не расходиться с `docs/`:

| Аспект | База `Ready` (as-is в `docs/`) | Оркестратор (это видение) |
|---|---|---|
| Главный артефакт | Готовое ТЗ / PRD, метка **Ready** | **Код → PR** (ТЗ становится промежуточным этапом) |
| Процесс | Один движок готовности, 4 контура | **Настраиваемый конвейер** из блоков под команду/тип задачи |
| Настройка | Правка `.md` руками, «никакого UI-конструктора в MVP» | **Админка-конструктор** генерит конфиг Ouroboros |
| Авто-PR | Намеренно выкинут ради демо (стретч) | **Ядро** ценности |

> Практический вывод (решение 02.07): **заявка продаёт оркестратор идея→PR целиком** — это
> видение и есть рамка Project Proposal. Прототип гарантирует этапы «сигнал → готовность →
> выгрузка» + охрану + эволюцию; этап кода — ранняя версия (happy-path до PR), скипается без
> потери ценности. Конструктор из блоков — за пределами MVP (настройка через `.md`). Звезда
> демо — по-прежнему «превращение сырой задачи → готовность». См. раздел «Границы».

---

## Опора 1. Оркестратор (SDD-конвейер)

**Слушает сигналы из любых каналов и запускает из них процесс.**

```
сигнал (канал)  ──►  завести/дополнить задачу  ──►  провести по этапам процесса  ──►  артефакт
```

- **Каналы-сигналы:** тег агента в чате (Slack / СберЧат), расшифровка встречи (**Jazz**),
  задача в Jira, любой контекст обсуждения. Из сигнала оркестратор сам заводит/дооформляет
  задачу по процессу команды.
- **Пример стандартного процесса** (настраивается): `Jira-таска → интервью → ТЗ → аналитика →
  дизайн → код → PR`.
- **Каждый этап явный, состояние — вне сессии LLM.** Оркестратор всегда знает, где задача, что
  сделано, какой PR, и может продолжить после падения/перезапуска (resume — ответственность
  системы). Это основа наблюдаемости и доверия.

---

## Опора 2. Конструктор процесса (админка)

Веб-панель, где из **блоков** собирается процесс под любую команду или тип задачи:

- **Этапы** — шаги конвейера (интервью, ТЗ, аналитика, дизайн, код, ревью…).
- **Роли** — кто отвечает за этап (аналитик, продакт, дизайнер, разработчик).
- **Гейты** — условия перехода между этапами (например, quality gate «зелёные тесты»,
  «критичные вопросы спеки закрыты»).
- **Границы автономии `Always / Ask / Never`** — что агент делает сам, что через апрув человека,
  что не делает никогда.

Нажатие **«Применить»** → система генерит конфиг **Ouroboros**: `SKILL.md` ролей + A2A-оркестрацию.
Процесс **не захардкожен** — он данные, а не код. Правка процесса = пересборка конфига из блоков.

> Связь с базой: `.md` остаётся источником правды процесса ([architecture.md](architecture.md)).
> Конструктор — это UI-надстройка, которая **редактирует те же `.md` визуально** и собирает из них
> конфиг оркестрации. Решение 02.07: конструктор целиком в **целевом видении** (вне MVP и вне
> обещаний заявки) — тезис architecture.md «никакого UI-конструктора в MVP» остаётся в силе.

Панель также служит **дашбордом**: по каждой задаче видно стадию, ветку, PR, результат проверок,
потраченную стоимость и следующее действие.

---

## Опора 3. Самоэволюция

Три момента жизненного цикла процесса:

1. **Инициализация (онбординг команды).** Оркестратор добавляет команду и создаёт процесс:
   расспрашивает о процессе, задачах, DoR/DoD; уточняет состав, роли, цели команды (возможный
   источник — релизный портал). Опирается на базу «хорошего процесса» + 5 эталонных задач +
   интервью (калибровка, см. [concept.md](concept.md)).
2. **Фиксация.** Закрепляет процесс под команду и **привязывает роли к этапам** конвейера.
3. **Ретро.** По сигналам из практики (что влетало, менялось, переоткрывалось) подсказывает, как
   улучшить SDD-процесс — какие **гейты** и **скилы** добавить. Выдаёт `diff` к процессу → апрув
   человека. Детали петли → [evolution.md](evolution.md).

## Инженерные инварианты оркестратора

Требования к системе, без которых оркестратору не будут доверять. Сгруппированы по цели.
Полный разбор → [engineering-principles.md](engineering-principles.md); ниже — формализованный список.

### A. Не начинать работу на мутной задаче
1. **Интервью-гейт перед кодом.** Агент-аналитик читает тикет и возвращает список проблем:
   противоречия в требованиях, недостающие состояния экрана (loading / error / empty / offline),
   неясные формулировки, скрытые зависимости. Пока критичные пункты не закрыты человеком —
   генерация кода **не стартует**. Поймать на спеке дешевле, чем переделывать PR.

### B. Надёжность процесса
2. **Явное состояние, сохраняемое вне сессии LLM.** Стадия, что сделано, какой PR — фиксируются
   отдельно. Упал прогон или перезапустили систему → оркестратор продолжает с того же места.
3. **Изоляция задачи в песочнице.** Каждая задача — своя ветка/worktree: параллельные задачи не
   мешают, легко diff и откат (чекпойнт `git diff --binary` → при ошибке `reset --hard`), чистка
   за собой. Нужно и для безопасности, и для диагностики.

### C. Доверие к результату
4. **«Готово» от агента не считается — проверяем независимо.** Отдельный quality gate гоняет
   тесты, линт, сборку (опц. LLM-ревью с порогом уверенности). MR открывается **только на зелёной
   проверке**, а не по заявлению агента.

### D. Контроль и бюджет
5. **Лимит на починку.** Цикл план → код → проверка → починка ограничен: максимум попыток,
   таймаут, бюджет токенов. Упёрся в лимит → агент останавливается и **зовёт человека**, а не
   чинит одну ошибку часами.
8. **Управляемость человеком-оператором.** Стоп / пауза / рестарт / уточнение задачи и видимая
   стоимость в любой момент. Оркестратор — система **под управлением человека**, а не «агент,
   который просто пишет код».

### E. Прозрачность
6. **Наблюдаемость с первого дня.** По каждой задаче видно: стадия, ветка, PR, результат проверок,
   стоимость, следующее действие. Не понимаешь, что внутри — не доверяешь.

### F. Экономный контекст
7. **Выборочный контекст и курированная память.** Агенту дают ровно нужный кусок (спека, карта
   модулей `AGENTS.md`, правила команды), а не весь репозиторий. Память между сессиями (решения,
   паттерны) — **курированная**, а не свалка.

## Роли и назначение на этапы

Механика движка одна, меняется «что значит готово» и кто отвечает. Конструктор привязывает роль
к этапу:

| Роль | Вход | Готово, когда… |
|---|---|---|
| Аналитик | сырая гипотеза / сигнал | ТЗ передаётся дальше без вопросов |
| Продакт | идея фичи | PRD согласован, метрики и scope зафиксированы |
| Дизайнер | задача на макет | бриф + критерии, готовые к проектированию |
| Разработчик | готовое ТЗ | **PR на зелёном quality gate** |

## Обратная связь команде (решено 02.07)

**Каналы агностичны:** сигнал на вход — из любого подключённого канала (тег в чате, встреча,
Jira, командная строка), обратная связь — в любой канал, куда Ouroboros умеет писать
(транспорты/MCP). **Дефолт аудируемости: комментарий в Jira-задаче + пинг автора** — вся
история в задаче, автор получает точечное уведомление. **Для демо — Jira + Telegram**
(telegram-bridge — штатный транспорт Ouroboros). Настройка каналов на команду — конструктор
(целевое видение).

Связанное поведение: **явный статус задачи** и **возврат репортеру** незаведённых/недооформленных
задач (перекликается с контуром «Охраняет»).

## Интеграции

- **Ouroboros** — движок оркестрации агентов (аналог Claude Code); скилл = `SKILL.md` ролей + A2A.
- **MCP Atlassian** — Jira (чтение/дозаполнение/поиск/комментарии, поиск дублей) + Confluence.
- **Jazz** — расшифровка встречи → задача по процессу.
- **Каналы связи (Telegram / Slack / СберЧат / …)** — захват обсуждения в задачу и обратные
  уведомления; агностично, любой транспорт/MCP Ouroboros. Для демо — Telegram.
- **Релизный портал** *(под вопросом)* — источник состава/ролей/целей команды при онбординге.

## Границы: полное видение vs хакатон-MVP

| Блок | Хакатон-MVP | Целевое видение |
|---|---|---|
| Конвейер | Захват → готовность (ТЗ) — «звезда» демо | Идея → PR сквозным графом |
| Конструктор | `.md` + минимальный дашборд | Полная админка из блоков + границы автономии |
| Артефакт | Готовое ТЗ + метка Ready | Код на зелёном quality gate |
| Самоэволюция | `diff` к процессу по сигналам, апрув человека | Живой стандарт + метрики процесса |
| Инженерные инварианты | Наблюдаемость, human-in-the-loop, интервью-гейт | Полный набор A–F, песочница, бюджет, resume |

Принцип питча остаётся: **маленькое и железное — показываем; большое и неизбежное — рассказываем.**
