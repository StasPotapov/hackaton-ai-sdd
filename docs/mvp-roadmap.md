# MVP-scope и слои фич

> Что делаем к демо, что стретч, что рассказываем как мечту. Принцип: **маленькое и железное —
> показываем, большое и неизбежное — рассказываем.**
>
> Рамка (решение 02–04.07): **коммитимся на полный цикл идея→PR**. Прототип целится в сквозной
> проход сигнал → готовность → выгрузка → код → PR плюс охрану и самоэволюцию; каждый этап — с
> апрувом человека и чек-поинтом. Этап кода на демо — отрепетированный happy-path прогон.

## Три главные фичи (звёзды демо, на них давим)

1. **Заводит задачи сам, по вашему процессу.** Тег агента в чате или встреча из **Jazz** → суть
   вытянута → готовая задача в Jira по стандарту команды.
2. **Охраняет входящие.** Мониторинг очереди Jira → сверка с процессом → авто-коммент автору о
   том, чего не хватает, с просьбой докинуть.
3. **Самоэволюция процесса.** Раз в спринт анализ задач команды (что влетало, как менялось,
   реопены) → предложение улучшить процесс → `diff` к `.md` → апрув человека.

## Ядро MVP (обязательный скоуп к 20.07)

Список зафиксирован синком 02–04.07 (§3 протокола; протокол — в git-истории):

0. **Самоэволюция** — ретро процесса команды по её SDD: `signals.log` (≥2 источника) → дайджест →
   `diff` к `.md` → апрув. См. [evolution.md](evolution.md).
1. **Калибровка-интервью** — база «хорошего процесса» + интервью с командой + скан источников →
   чек-лист и шаблоны в `.md`. См. [concept.md](concept.md).
2. **Работа с MCP Atlassian + Telegram** — чтение/запись Jira и Confluence (label `Ready`),
   сигналы и пинги в канал.
3. **Полный процесс идея → PR (SDD)** — готовность (черновик → самокритика по чек-листу →
   вопросы человеку) → аналитика → дизайн → код → PR на зелёном quality gate.
4. **Чек-поинты (стейт-машина)** — состояние каждого этапа сохраняется вне сессии LLM; можно
   вернуться к любому шагу и продолжить (устойчивость к обрыву модели).
5. **Дашборд + мониторинг** — вкладка в web-UI Ouroboros: мониторинг выполненных задач, логи с
   метриками (здоровье pipeline).
6. **Классификатор задачи** — баг / фича / … → свой процесс под тип задачи.

## Слой 1 — приятные усилители (если успеем)

| Фича | Почему полезно | Почему дёшево |
|---|---|---|
| **Три вопроса сеньора** — ровно 3 самых больных уточнения, не 20 | мышление сеньора, не форма | уже вшито в калибровку-интервью |
| **«Это дубль!»** — находит похожую/дублирующую задачу | момент «откуда он знает?» | поиск по Jira через MCP |
| **DoR-линтер на стенде** — вставь задачу → «готовность 40%, нет критериев приёмки» | прохожий становится фанатом за 5 сек | срез процесса без полного пайплайна |
| **Readiness score** для наглядности готовности | видимая полоска прогресса | побочный продукт самокритики |
| **Доска процесса** — kanban задач по этапам поверх дашборда ядра (`ui_tab`) | наблюдаемость становится продуктом | `ui_tab`-виджет скилла, свой фронт не нужен |
| **Стоимость задачи** — видимая цена токенов per задача/этап + бюджет-лимит | «видишь, во что обходится агент» — доверие и управляемость затрат | Dashboard→Costs и учёт токенов — штатные в Ouroboros |

## Слой 2 — стретч (если останется время)

- Нотификация (Slack/СберЧат/почта), когда задача стала **Ready**.
- Возврат недооформленной задачи репортёру.
- Захват обсуждения фичи в Slack/СберЧате → задача в обратку.

## Переехало из «мечты» в «делаем» (решение синка 04.07)

- **Сквозная нить** идея → … → релиз — делаем на дашборде (ядро п.5).
- **Импакт-анализ** «меняем требование — что задето» — через классификатор задачи (ядро п.6).
- **Живой стандарт SDD** — стандарт зашит, эволюция дописывает правила (петля самоэволюции).

## Слой 3 — будущее (рассказываем со сцены как мечту)

- **Оркестрация ролей:** задача идёт по эстафете аналитик → продакт → дизайнер → dev, каждому —
  готовый вход.
- **Онбординг новичка:** «как у нас принято писать ТЗ?» — на ваших же примерах.
- **UI-конструктор из блоков:** из блоков (этапы, роли, гейты, границы Always/Ask/Never)
  собираешь процесс → «Применить» → конфиг Ouroboros. Прорабатывается отдельной веткой (Ваня),
  в гарантированный скоуп MVP не входит; в MVP настройка только через `.md`.

## Осознанно НЕ делаем в MVP (решение синка 04.07)

- **Story points по аналогии.**
- **Устойчивый этап кода на произвольных задачах** (за пределами happy-path) — в демо он
  появляется как один отрепетированный happy-path прогон до PR.

## Зафиксированные решения (02–04.07)

- **Каналы связи — агностичны:** вход из любого подключённого канала (чат, встреча, Jira,
  командная строка), выход — в любой транспорт/MCP Ouroboros. Дефолт аудируемости — комментарий
  в Jira-задаче + пинг автора. **Для демо — Jira + Telegram** (telegram-bridge — штатный
  транспорт Ouroboros).
- **Звезда демо** — весь конвейер идея→PR **и** фаза ретро/самоэволюции (показываем оба).
