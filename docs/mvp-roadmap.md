# MVP-scope и слои фич

> Что делаем к демо, что стретч, что рассказываем как мечту. Принцип: **маленькое и железное —
> показываем, большое и неизбежное — рассказываем.**
>
> Рамка (решение 02.07): **продаём конвейер идея→PR целиком**; прототип гарантирует этапы
> «сигнал → готовность → выгрузка» + охрану + эволюцию, этап кода — ранняя версия (happy-path
> до PR), скипается в любой момент без потери ценности.

## Три главные фичи (звёзды демо, на них давим)

1. **Заводит задачи сам, по вашему процессу.** Тег агента в чате или встреча из **Jazz** → суть
   вытянута → готовая задача в Jira по стандарту команды.
2. **Охраняет входящие.** Мониторинг очереди Jira → сверка с процессом → авто-коммент автору о
   том, чего не хватает, с просьбой докинуть.
3. **Самоэволюция процесса.** Раз в спринт анализ задач команды (что влетало, как менялось,
   реопены) → предложение улучшить процесс → `diff` к `.md` → апрув человека.

## Ядро MVP (фундамент под фичи)

1. **Калибровка с интервью** — база «хорошего процесса» + 5 задач + доспрос → чек-лист и шаблоны
   в `.md`. См. [concept.md](concept.md).
2. **Процесс готовности** — захват → черновик → самокритика по чек-листу → уточняющие вопросы.
3. **Захват в Jira-задачу** (из чата/встречи) + **выгрузка в Confluence** + label `Ready`.
4. **Мониторинг очереди → авто-коммент** (фича 2).
5. **Онбординг Q&A.**
6. **Петля эволюции (минимум):** `signals.log` (≥2 источника: реопены + доправки) + дайджест +
   `diff` к `.md`. См. [evolution.md](evolution.md).
7. **Этап кода (ранняя версия):** готовое ТЗ → изолированная ветка/worktree → код → quality gate
   → черновик PR. Один отработанный happy-path прогон.

## Слой 1 — приятные усилители (если успеем)

| Фича | Почему полезно | Почему дёшево |
|---|---|---|
| **Три вопроса сеньора** — ровно 3 самых больных уточнения, не 20 | мышление сеньора, не форма | один промпт поверх процесса |
| **«Это дубль!»** — находит похожую/дублирующую задачу | момент «откуда он знает?» | поиск по Jira через MCP |
| **DoR-линтер на стенде** — вставь задачу → «готовность 40%, нет критериев приёмки» | прохожий становится фанатом за 5 сек | срез процесса без полного пайплайна |
| **Readiness score** для наглядности готовности | видимая полоска прогресса | побочный продукт самокритики |
| **Доска процесса** — вкладка в web-UI Ouroboros: задачи по этапам, гейты, следующее действие *(кандидат 02.07, обсуждается — DISCUSSION.md 2.4)* | наблюдаемость становится продуктом | `ui_tab`-виджет скилла, свой фронт не нужен |
| **Стоимость задачи** — токены/деньги per задача/этап + бюджет-лимит *(кандидат 02.07, обсуждается — DISCUSSION.md 2.5)* | «видишь, во что обходится агент» — доверие + факты для финмодели | Dashboard→Costs и учёт токенов — штатные в Ouroboros |

## Слой 2 — стретч (если останется время)

- Доведение этапа кода до устойчивости (за пределами happy-path: правки по ревью, повторные прогоны).
- **Оценка по аналогии** — story points по похожим прошлым задачам (усиливает ров).
- Нотификация (Slack/СберЧат/почта), когда задача стала **Ready**.
- Возврат недооформленной задачи репортёру.
- Захват обсуждения фичи в Slack/СберЧате → задача в обратку.

## Слой 3 — будущее (рассказываем со сцены как мечту)

- **Сквозная нить (оркестратор):** идея → PRD → ТЗ → задачи → PR → релиз одним графом. Отвечает
  на «почему эта строчка кода существует?» за секунду.
- **Импакт-анализ:** «меняем требование — вот какие задачи/код/тесты задеты». Две недели
  совещаний → один запрос.
- **Живой стандарт:** новое замечание на ревью → скилл предлагает дописать правило (расширение
  петли эволюции).
- **Оркестрация ролей:** задача идёт по эстафете аналитик → продакт → дизайнер → dev, каждому —
  готовый вход.
- **Метрики процесса:** дашборд — где задачи застревают, среднее время до Ready, узкие места.
- **Онбординг новичка:** «как у нас принято писать ТЗ?» — на ваших же примерах.
- **Админка-конструктор:** из блоков (этапы, роли, гейты, границы Always/Ask/Never) собираешь
  процесс под любую команду/тип задачи → «Применить» → генерит конфиг Ouroboros. Не
  захардкоженный пайплайн, а настраиваемый.

## Зафиксированные решения (02.07)

- **Каналы связи — агностичны:** вход из любого подключённого канала (чат, встреча, Jira,
  командная строка), выход — в любой транспорт/MCP Ouroboros. Дефолт аудируемости — комментарий
  в Jira-задаче + пинг автора. **Для демо — Jira + Telegram** (telegram-bridge — штатный
  транспорт Ouroboros).
- **Конструктор из блоков:** целевое видение (Слой 3), в MVP настройка только через `.md`.

## Открытые вопросы (нужно прогрумить)

- **Jazz-интеграция** — детали.
- **Сбор инфы перед интервью** при настройке процесса по Jira-таскам.

## Что осознанно НЕ обещаем к демо

Устойчивый этап кода на произвольных задачах — в демо он появляется как один отрепетированный
happy-path прогон до PR (или одной фразой, если не дозреет). Звезда демо — превращение сырой
задачи в готовое ТЗ; конвейер при этом продаём целиком: этапы независимы, деградация до ТЗ не
теряет ценности.
