# Видение продукта: что хотим получить в итоге

> Главный документ базы знаний. Отвечает на «зачем, что, как» — и описывает конечную цель,
> к которой сходятся все остальные доки.

## Суть в одном предложении

**«Поговори со мной — и я покажу тебе твой процесс: расспрошу команду, как опытный тимлид на
онбординге, и сверю ответы с вашими задачами, чатами и вики. А потом проведу любую задачу по
этому процессу — от сигнала до PR, останавливаясь там, где скажешь.»**

Это не генератор документов и не ещё один код-агент. Это **самоэволюционирующийся
SDD-оркестратор**: сигнал (Jira-задача, тег в чате, встреча) входит → задача идёт по этапам
конвейера команды (готовность/ТЗ → аналитика → дизайн → код) до PR на зелёном quality gate.
Первый и главный гейт конвейера — **готовность (Definition of Ready)**: сырая задача → чёткое
ТЗ без единого «а что ты имел в виду?». Каждый этап — самостоятельная ценность, конвейер можно
остановить на готовом ТЗ.

## Проблема (настоящая боль)

В компании все хотят SDD (Spec-Driven Development), но никто не может описать, как он выглядит.
Поэтому:
- регламенты на 40 страниц никто не читает;
- внедрение буксует на «давайте сначала договоримся о процессе»;
- задачи уходят в разработку сырыми → переделки, возвраты, потерянное время.

**Ключевая идея:** процесс не описываем — **добываем из команды: интервью + её живые источники.**
Порог входа стремится к нулю.

## Что мы хотим получить (конечное состояние)

Один скилл — **живой носитель процесса команды** поверх Ouroboros и MCP Atlassian, который:

1. **Добывает** ваш стандарт через **калибровку с интервью**: скилл не с нуля — в него заложена
   база «каким должен быть хороший процесс». Главный источник — **интервью с командой**: скилл
   расспрашивает, как опытный тимлид на онбординге. Параллельно **сканирует все доступные
   источники** (бэклог Jira, Confluence, чаты) — не как эталон, а как свидетельства практики:
   сверяет «как говорим» с «как делаем», расхождения превращает в вопросы. Команде без истории
   хватает базы + интервью. Итог → чек-лист готовности и шаблоны в `.md`-файлах (это и есть вся
   «настройка» — правишь markdown, не код и не UI).
2. **Применяет** этот стандарт: ловит сигнал (Jira, чат, Jazz), ведёт задачу по этапам конвейера
   (захват → черновик → самокритика по чек-листу → уточняющие вопросы человеку → выгрузка →
   следующие этапы вплоть до кода и PR). Кладёт готовое ТЗ в Confluence, дозаполняет Jira,
   ставит метку **Ready**; на этапе кода — изолированная ветка, независимый quality gate, PR.
3. **Охраняет** процесс: мониторит очередь, ловит дубли, возвращает недооформленные задачи
   репортёру.
4. **Учит** процессу: онбординг новичка отвечает «как у нас принято писать ТЗ?» на реальных
   примерах команды.
5. **Эволюционирует**: раз в спринт смотрит на задачи команды → предлагает `diff` к `.md`
   (добавить/убрать пункт, поменять шаблон) → человек апрувит. Процесс не мёртвый.

## Три главные фичи (на них давим в первую очередь)

1. **Заводит задачи сам, по вашему процессу.** Тегни агента в переписке в чате или дай ему
   встречу из **Jazz** — он вытащит суть из обсуждения/расшифровки и откроет готовую задачу в
   Jira сразу по вашему стандарту (нужные поля, структура, критерии). Он уже знает ваш процесс.
   Обсуждение → задача в один тег.
2. **Охраняет входящие задачи.** Мониторит очередь Jira: кто-то завёл задачу — скилл сверяет её
   с процессом и, если чего-то не хватает (описание, макет/картинка, критерии приёмки, метрика),
   **сам пишет комментарий автору** и просит докинуть. Недооформленное не идёт дальше.
3. **Самоэволюция процесса** (то, что хотят услышать жюри). Раз в спринт смотрит на задачи
   команды — какие влетали, как их описывали, как они менялись, как часто переоткрывались — и на
   этих знаниях **предлагает, чем улучшить процесс**. Меняет не сам: `diff` к `.md` → апрув
   человека. Прозрачное, объяснимое самосовершенствование.

**Единый движок — все роли.** Меняется только чек-лист «что значит готово»:

| Роль | Вход | Выход (готово, когда…) |
|---|---|---|
| Аналитик | сырая гипотеза | ТЗ передаётся в разработку без вопросов |
| Продакт | идея фичи | PRD согласован, метрики и scope зафиксированы |
| Дизайнер | задача на макет | бриф + критерии, готовые к проектированию |
| Разработчик | готовое ТЗ | PR на зелёном quality gate (этап кода конвейера) |

## Почему это выигрывает (ров проекта)

| Слабое место конкурентов | Чем закрываем |
|---|---|
| «Это же просто ChatGPT пишет PRD» | Стандарт добыт из **вашей** команды (интервью + все источники) + замкнутая петля самокритики. Генерацию покажут все — это никто. |
| «Дайте гибкую базу» = пустота | База — это **мнение**: жёсткий формат, мягкое содержание. Спорить с конкретным = находить свой процесс. |
| «Процесс» звучит скучно для жюри | **Превращение на сплит-экране** + ползущая метрика readiness 25%→95%. |
| «Только для разработчиков» | Один примитив, две роли на сцене — продаёт менеджменту. |
| Высокий порог внедрения | **Самонастройка вживую**: одно интервью + автоскан источников → ноль внедрения. |

## Принцип питча

**Маленькое и железное — показываем. Большое и неизбежное — рассказываем.**
Связка одной фразой: *«Сегодня он доводит задачу до готовности. Завтра — держит нить от идеи
до релиза.»*

## Границы: что продаём и что показываем

**Продаём конвейер целиком** (идея → PR) — это рамка заявки. **Показываем железное:** звезда
демо — превращение сырой задачи в готовое ТЗ; этап кода — ранняя версия, один happy-path прогон
до PR, включаем в показ ровно настолько, насколько он железный. Конвейер спроектирован так, что
этап кода **скипается в любой момент без потери ценности** — управляемая деградация, а не
срезание скоупа. Конструктор процесса из блоков — целевое видение, в MVP настройка только через
`.md` (см. [orchestrator.md](orchestrator.md)).
