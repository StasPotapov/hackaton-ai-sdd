# Архитектура: как это устроено

> Технический контур решения. Как движок, `.md`-файлы, MCP и human-in-the-loop складываются
> в работающую систему поверх Ouroboros.

## Ключевой принцип: `.md` — единственный источник правды

Процесс команды живёт не в коде и не в БД, а в markdown-файлах скилла:

- `checklist.md` — чек-лист готовности (что значит «готово») для роли/типа задачи.
- `template.md` — шаблон артефакта (ТЗ / PRD / бриф / аналитика).
- Возможны подвиды: `bug.template.md`, `feature.template.md` и т.п.

Все 4 контура (Захват / Готовность / Охрана / Онбординг) читают одни и те же `.md`. Петля
эволюции — единственный, кто предлагает их менять (через `diff` + апрув человека). Правка
процесса = правка markdown. Никакого UI-конструктора в MVP не требуется — конструктор из блоков
остаётся в целевом видении ([orchestrator.md](orchestrator.md)).

## Платформа и интеграции

- **Ouroboros** — движок оркестрации агентов (аналог Claude Code). Наш скилл — тип `extension`:
  манифест `SKILL.md` + `plugin.py` (PluginAPI). Штатные механизмы платформы, которые используем:
  - **`scheduled_tasks` (cron) в манифесте скилла** — планировщик Ouroboros сам будит агента по
    расписанию → фоновый мониторинг очереди Jira (фича «охраняет») и спринтовый дайджест эволюции;
  - **встроенный MCP-клиент** (настройка `MCP_SERVERS`, транспорт streamable HTTP/SSE) —
    подключение Atlassian без своего кода интеграции;
  - **субагенты** с model lanes (main/heavy/light/review) под вес этапа и изоляцией
    acting-субагента в **git-worktree** с проверяемой интеграцией патча — этап кода из коробки;
  - **Dashboard** (Logs / Costs / Activity) — наблюдаемость и учёт стоимости токенов на задачу.
- **MCP Atlassian** — Jira (чтение/дозаполнение задач, поиск, комментарии) + Confluence
  (выгрузка ТЗ/PRD, метки).
- **Каналы связи — агностичны** (Telegram / Slack / СберЧат / командная строка): захват
  обсуждений в задачу (фича 1) и обратные уведомления — любой транспорт/MCP, куда Ouroboros
  умеет писать. Для демо — **Telegram** (telegram-bridge — штатный transport-скилл Ouroboros).
- **Jazz** — источник для фичи 1: расшифровка встречи → задача по процессу.

## Калибровка через интервью (контур «Добывает»)

Скилл не с нуля — в него заложена **база «каким должен быть хороший процесс»** (обязательные
атрибуты задачи, этапы). Главный инструмент — **интервью**: скилл расспрашивает команду, как
опытный тимлид на онбординге (продукт, роли, этапы, «что значит готово», где болит). Параллельно
**сканирует все доступные источники** (бэклог Jira, Confluence, чаты) — как свидетельства
практики, не эталон: сверяет ответы с тем, как задачи ведутся на самом деле, расхождения
превращает в вопросы. Итог пишет в `checklist.md` / `template.md`. Команде без истории хватает
базы + интервью — эволюция дозатачивает стандарт по мере накопления задач.

```
база «хорошего процесса»     скан источников команды
 (атрибуты, этапы)           (Jira / Confluence / чаты)
            │                        │
            └───────────┬────────────┘
                        ▼
        интервью с командой — главный источник
        (вопросы + сверка «говорим ↔ делаем»)
                        ▼
   checklist.md + template.md  (команда правит руками)
```

## Движок применения (процесс готовности одной задачи)

```
задача (из Jira / заведена из чата или Jazz)
      │  MCP: get task + связанные + эпик + похожие; проверка на дубли
      ▼
  [Захват] контекст собран
      ▼
  draft ТЗ по template.md
      ▼
  самокритика по checklist.md → чего не хватает
      ▼
  уточняющие вопросы человеку (human-in-the-loop)
      ▼
  готовое ТЗ
      ▼
  [Выгрузка] Confluence + Jira дозаполнение + метка Ready
      ▼
  [Этап кода] изолированная ветка/worktree → код → независимый quality gate
      ▼
  PR (только на зелёной проверке; этап включается по готовности, скипается без потери ценности)
```

## Human-in-the-loop

На каждом стыке — **«дальше / поправить / стоп»**. Это не чёрный ящик, а конвейер с чекпоинтами.
Скилл никогда не меняет процесс сам — только предлагает. Каждое изменение видно, причина видна,
апрув человека обязателен → «прозрачная самоэволюция».

## Инженерные требования оркестратора

Детально → [engineering-principles.md](engineering-principles.md). Кратко, что должна обеспечивать
система (важно для доверия и для критерия «Процесс работы агента»):

1. **Не кодить на мутной задаче** — агент-аналитик сначала возвращает список проблем спеки;
   генерация не стартует, пока критичные пункты не закрыты человеком.
2. **Явное состояние вне сессии LLM** — стадия, PR, что сделано фиксируется отдельно; resume —
   ответственность системы, а не «повезёт с контекстом».
3. **Изоляция в песочнице** — каждая задача в своей ветке/worktree; чекпойнт `git diff --binary`,
   откат `reset --hard`, чистка за собой.
4. **Независимый quality gate** — «готово» агента не принимается на веру; тесты/линт/сборка
   (+ опц. LLM-ревью с порогом уверенности); MR только на зелёной проверке.
5. **Лимит на починку** — макс. попыток, таймаут, бюджет токенов; упёрся → зовёт человека.
6. **Наблюдаемость с первого дня** — по каждой задаче: стадия, ветка, PR, результат проверок,
   стоимость, следующее действие.
7. **Выборочный контекст и курированная память** — ровно нужный кусок (спека, карта модулей
   `AGENTS.md`, правила команды), а не весь репозиторий.
8. **Управляемость человеком** — стоп/пауза/рестарт/уточнение и видимая стоимость в любой момент.
9. **Детерминированное — кодом, LLM — только семантика** (принцип синка 9.9): проверки наличия
   полей/меток/секций — код в `plugin.py`; LLM подключается там, где нужна семантика («критерии
   есть, но неизмеримые»). Дёшево по токенам (лимит $300), не флапает на демо; по этому принципу
   строятся охранник очереди и DoR-линтер.

## Данные и состояние

- Сигналы эволюции копятся в `signals.log` (см. [evolution.md](evolution.md)).
- Состояние задач — вне контекста LLM (для resume и наблюдаемости).
- Память между сессиями — **курированная** (принятые решения, паттерны), не свалка.
