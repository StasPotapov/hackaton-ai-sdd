# Критерии оценки Project Proposal и как мы их закрываем

> Рабочая шпаргалка для подготовки заявки. По каждому критерию — что требует жюри и чем отвечаем.
> Итоговая оценка отборочного этапа — средневзвешенная (макс. 5 баллов).

## 1. Описание кейса — вес 30% (макс. 5 баллов)

**Что требуется:** название и описание инициативы в формате **«зачем, что, как?»**; решаемая
проблема; детали для понимания сути.

**Чем закрываем:** [vision.md](vision.md) + [orchestrator.md](orchestrator.md) — SDD-оркестратор
идея→PR: боль (все хотят SDD, никто не описывает; код-агенты стартуют на мутных задачах),
решение (процесс добываем из команды — интервью + скан источников, конвейер с гейтами и human-in-the-loop, остановка на
любом этапе), 5 глаголов, единый движок для всех ролей. Это самый весомый критерий — раскрыть
максимально ясно.

## 2. Эффект от внедрения — вес 20%

**Что требуется:** ожидаемый финансовый эффект; доходная и расходная составляющие; детализация
по статьям; расчёт **Value to Cost**; за счёт чего достигается эффект; бизнес-метрики в формате
**AS IS → TO BE**.

**Чем закрываем:**
- **AS IS:** сырые задачи → возвраты, переделки, «а что ты имел в виду?», недели на согласование
  процесса, регламент на 40 страниц никто не читает.
- **TO BE:** задача проходит DoR за один прогон, метка Ready, ноль возвратов по «непонятно»,
  PR только на зелёном quality gate.
- **За счёт чего:** снижение переделок PR (поймали на спеке дешевле), сокращение времени
  аналитика/тимлида на груминг, снижение времени онбординга.
- **Метрики (качественно, ориентиры без опоры на количественный замер):** динамика времени до
  готовности (метка Ready), доля Reopened-задач, кол-во уточняющих итераций, итераций до зелёного
  MR, время онбординга новичка.
- Эффект подан качественно: качество и предсказуемость, меньше возвратов, AI-adoption
  (деньги/ROI сняты 04.07).

## 3. Процесс работы агента — вес 15%

**Что требуется:** пошаговое описание работы агента; как встраивается в текущий бизнес/тех-процесс.

**Чем закрываем:** движок из 4 шагов ([concept.md](concept.md)) + инженерные принципы
([engineering-principles.md](engineering-principles.md)): human-in-the-loop, quality gate, resume,
песочница, лимиты. Встраивание — поверх существующих Jira/Confluence, порог внедрения ~ноль.

## 4. Функции прототипа (MVP) — вес 15%

**Что требуется:** ожидаемый функционал на стадии прототипа к основному этапу.

**Чем закрываем:** [mvp-roadmap.md](mvp-roadmap.md) — три главные фичи (заводит задачи сам из
чата/Jazz; охраняет входящие авто-комментом; самоэволюция процесса раз в спринт) + ядро
(калибровка-интервью, полный SDD идея→PR, чек-поинты, дашборд+мониторинг, классификатор,
самоэволюция) + усилители (3 вопроса, дубли, DoR-линтер). Рамка: конвейер целиком, этапы
независимы.

## 5. Ограничения — вес 10%

**Что требуется:** регуляторные ограничения; источники данных; ограничения кибербезопасности и
других регуляторов.

**Чем закрываем (нужно проработать):**
- **Источники данных:** Jira, Confluence, Slack/СберЧат (через MCP/интеграции).
- **Кибербезопасность:** данные задач — внутренний контур Банка; работа через OpenRouter на
  основном этапе (ключи с лимитом) → вопрос передачи чувствительных данных вовне; изоляция в
  песочнице; апрув человека на все изменения.
- **Регуляторика:** банковская тайна, ПДн в задачах — требует явной оговорки. ⚠️ Проработать.

## 6. Реализуемость — вес 10%

**Что требуется:** пошаговый процесс реализации прототипа на Ouroboros; потенциал
масштабируемости в подразделении; переиспользование другими подразделениями; риски внедрения и
их нивелирование.

**Чем закрываем:**
- **Реализация на Ouroboros — конкретными примитивами платформы:** скилл типа `extension`
  (`SKILL.md` + `plugin.py`), `scheduled_tasks` (cron) для фонового мониторинга, встроенный
  MCP-клиент для Atlassian, субагенты с изоляцией в git-worktree для этапа кода, встроенный
  Dashboard для наблюдаемости/стоимости. См. [architecture.md](architecture.md).
- **Масштабируемость:** единый движок, меняется только `.md` → тот же скилл для любой команды.
- **Переиспользование:** «интервью + скан источников» работает для любого подразделения без кода.
- **Риски:** слабые источники / нет истории (новая команда); чувствительные данные; принятие командой →
  нивелируем human-in-the-loop, прозрачным diff, петлёй эволюции.
