# Формулировка идеи: «Живой конвейер»

> ⚠️ **АРХИВ (решение 04.07).** Ранняя формулировка идеи; рамка частично устарела: роли
> «Аналитик → Dev → QA» и «пульт с пресетами S/M/L» не соответствуют финальной версии
> (этапы: готовность → аналитика → дизайн → код; настройка в MVP — правка `.md`).
> Актуальное — [../PROPOSAL.md](../PROPOSAL.md), [vision.md](vision.md), [concept.md](concept.md).

> Каноничная формулировка проекта в четырёх уровнях: **коротко → шире → фичи → инженерный
> контур («о чём мы подумали»)**. Собрана просеиванием нагенеренного материала
> (`deck` / `effect-model` / `agent-process` / `scenario`) через сито честности.
>
> **Правило файла:** здесь нет ни одной выдуманной цифры. Эффект описываем качественно, без ROI
> и денежных оценок. Публичные кейсы — только проверяемые.

---

## 1. Коротко (одно предложение)

> **«Живой конвейер» — самоэволюционирующийся SDD-оркестратор: ведёт задачу из Jira до Merge
> Request силами ролей-агентов под контролем человека — и сам предлагает, как улучшить процесс
> команды.**

Три слова для жюри: **SDD · самоэволюция · банковский контур.**
Одной фразой: *«Не ещё один код-агент, а SDD-процесс, который подстраивается под команду и
умнеет — прозрачно и под апрувом человека».*

---

## 2. Шире (абзац)

Готовые код-агенты стартуют на мутных задачах и выдают правдоподобный, но неверный код, который
дольше проверять, чем писать заново. А в банке агенту нельзя бесконтрольно менять репозиторий —
каждое действие должно быть объяснимо аудитору. «Живой конвейер» — это **не ещё один генератор
кода, а конвейер, который держат под контролем и который учится**. Он ловит задачу из канала
(Jira; тег в чате / созвон — как сигнал), проводит её через SDD-процесс *вашей* команды силами
трёх ролей (**Аналитик → Dev → QA**), доводит до Merge Request под тремя видимыми стопами
контроля — и раз в N задач приносит человеку дайджест «вот что я заметил» с конкретной правкой к
описанию процесса. Человек жмёт **Apply** — и следующая похожая задача ловит проблему сама, ещё
на входе. Ключевое: эволюционирует **описание процесса (`.md`), а не модель агента** — поэтому
для банка это не риск, а усиление контроля (у каждого изменения есть diff, причина, апрувер и
откат).

**Целевой пользователь:** команда разработки в банке (Сбер / ДЗО) + режим «на одного».
**Целевой артефакт — код.**

---

## 3. Основные фичи

| # | Фича | Суть |
|---|---|---|
| 1 | **Онбординг под команду** | Интервью про процесс / роли / DoR → фиксирует SDD-процесс именно этой команды (этапы + роли). Есть режим «на одного» |
| 2 | **Оркестратор Jira → PR** | Три роли-агента ведут задачу от тикета до Merge Request; человек касается входа (спека) и выхода (мёрж) |
| 3 | **⭐ Самоэволюция процесса** | Копит сигналы прогонов → дайджест + `diff` к `.md` → человек **Apply / Edit / Reject**. Это дифференциатор — активной human-gated самоэволюции процесса нет у аналогов |
| 4 | **Вход из каналов** | Задача из Jira через MCP-мост; тег в чате / созвон — как сигнал-триггер |
| 5 | **Пульт оператора** | Сборка процесса из пресетов S / M / L, live-карточка задачи, human-in-the-loop через Telegram |

## 4. Инженерный контур — «о чём мы подумали»

Не фичи для галочки, а осознанные инженерные решения. Именно они дают **доверие** и продают
управляемость (а не скорость) — самое ценное в банковском контуре. Каждый пункт защитим на доске.

- 🛑 **Три видимых стопа контроля.**
  - *Clarify* — на мутной задаче код не стартует, пока человек не принял критерии приёмки.
  - *Ask* — опасное действие (запись, `exec`, `git push`) уходит в Approval Gate.
  - *Bounded-loop* — исчерпан лимит попыток → эскалация человеку, а не бесконечный цикл.
- 🧰 **Изоляция задачи в песочнице.** Каждая задача — свой git-worktree; чекпойнт
  `git diff --binary`, откат `reset --hard`, чистка за собой. Параллельные задачи не мешают,
  легко посмотреть дифф и откатить. Нужно и для безопасности, и для диагностики.
- 💰 **Контроль бюджета.** Цикл план → код → проверка → починка ограничен: максимум попыток +
  таймаут + бюджет токенов. Упёрся в лимит — агент останавливается и зовёт человека, а не чинит
  одну ошибку часами и не жжёт бюджет токенов впустую.
- ✅ **Независимый quality gate.** «Готово» от агента не принимается на веру: тесты / линт /
  сборка (+ опц. LLM-ревью с порогом уверенности). MR открывается только на зелёной проверке.
- 🔒 **Границы Always / Ask / Never.** *Always* — чтение, `ls`, `git status` без спроса; *Ask* —
  запись / `exec` / `git push` через Approval Gate; *Never* — `DROP`, `rm -rf`, `curl` вне
  allowlist — заблокировано на уровне платформы, промпт-инъекция границу не обходит.
- 💾 **Явное состояние + resume.** Стадия / PR / что сделано фиксируются вне сессии LLM. Упал
  прогон или перезапустили систему — оркестратор продолжает с того же места. Resume —
  ответственность системы, а не «повезёт с контекстом».
- 🧠 **Выборочный контекст + курированная память.** Агенту дают ровно нужный кусок (спека, карта
  модулей `AGENTS.md`, правила команды), а не весь репозиторий. Память между сессиями (решения,
  паттерны) — курированная, не свалка.
- 👤 **Управляемость человеком.** Стоп / пауза / рестарт / уточнение задачи и видимая стоимость
  в любой момент. Оркестратор — система под управлением человека, а не «агент, который просто
  пишет код».

## Эффект

Продаём **качество и предсказуемость**, а не скорость инференса — эффект работает и на слабом
железе. Эффект — меньше возвратов и переделок PR, единый процесс вместо разнобоя личных стилей,
аудируемый след решений, рост AI-adoption в команде. Без ROI и денежных оценок — эффект подаётся
качественно, а не в рублях.

## Отстройка и честные границы (чтобы не порвали на защите)

- **Дифференциатор** = связка: настройка SDD под команду (онбординг) + банковский контур
  (Always/Ask/Never + аудит) + **самоэволюция процесса под человеком**. Линейный конвейер
  `тикет → спека → код → тесты → PR` — table stakes, умеют все.
- **Не «настоящий A2A».** На локальном MVP субагентный рой = центральный оркестратор. Говорим
  честно: «оркестрованная передача ролей», а не «магический протокол A2A».
- **Граница безопасности эволюции.** Правим только `.md` процесса, не код агента (`/evolve`
  Ouroboros не включаем). Критичное (гейты / safety / scope) — только через апрув с откатом.
- **Реализуемость.** Единственный наш код платформенного уровня — **MCP-мост к трекеру** (2–3
  инструмента). Роли и гейты — конфигурация и промпты (`SKILL.md`), остальное из коробки Ouroboros.
- **Публичные кейсы — только проверяемые.** Класс инцидента «автономный агент снёс прод»
  иллюстрируем реальным публичным случаем (напр. удаление продовой БД AI-агентом Replit, лето
  2025) — **точную формулировку сверить перед декой**. Выдуманные инциденты (в черновике был
  «PocketOS, апр. 2026») не используем.
