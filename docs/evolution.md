# Самоэволюция процесса (петля обратной связи)

> Закрывает требование хакатона о **самосовершенствовании** — элегантно: не «модель сама
> дообучается» (непрозрачно и страшно), а **процесс команды эволюционирует через наблюдения
> скилла**. Контролируемо, объяснимо, ценно.

## Концепция

**Задача скилла — делать процесс вашей команды лучше от спринта к спринту.** Раз в спринт он
смотрит на задачи команды целиком: какие влетали, как их описывали, как они менялись, как часто
их переоткрывали — и на этих знаниях предлагает, чем улучшить процесс.

Эволюция — **не пятый отдельный канал, а нервная система всего продукта**. Все 4 контура сливают
сигналы → скилл анализирует → предлагает изменения к `.md` → человек апрувит.

```
            checklist.md / template.md
                       ▲
                       │ apply (после approve)
                       │
        ┌──────────────┼──────────────┐
        ▼              ▼              ▼
    Захват  Готовность  Охрана  Онбординг
        │       │          │        │
        └───────┴─ сигналы ─┴────────┘
                       │
                       ▼
              ┌─ Эволюция ─┐
              │ дайджест + │
              │  diff к .md│
              └────────────┘
                       │
                       ▼
                Human review
```

## Какие сигналы собираем

| Источник | Сигнал | Что говорит |
|---|---|---|
| Доска Jira | Задачи, движущиеся назад (Reopened, обратно в Backlog) | Что-то не поймано в DoR |
| Контур готовности (DoR-loop) | Какие пункты чек-листа чаще «жёлтые» | Слабое место процесса |
| Охрана | Какие комменты скилла отклонили/проигнорили | Пункт не важен команде — кандидат на удаление |
| Захват | Какие поля задачи команда дополняет вручную после создания | Этого не хватает в шаблоне |
| Общее | Время прохождения через DoR | Тренд: процесс ускоряется/замедляется |

## Что скилл предлагает менять

- ➕ **Добавить пункт** в чек-лист — «8 из 10 возвращённых задач не имели метрики — добавить
  пункт "метрика продукта"?»
- ➖ **Убрать пункт** — «за 2 недели пункт "архитектурное обсуждение" не заполнили ни в одной
  задаче — удалить?»
- 🔄 **Изменить шаблон** — «команда всегда добавляет раздел "влияние на QA" вручную — встроить?»
- 🌿 **Подвид шаблона** — «у багов своя структура — выделить `bug.template.md`?»

## Механика — раз в N задач или в спринт

1. Скилл копит наблюдения в `signals.log`.
2. По расписанию (раз в неделю / каждые 20 задач) генерит **дайджест**: «вот что я заметил».
3. Делает **`diff` к `.md`** (как PR-предложение).
4. Тимлид жмёт **apply / reject / edit**.
5. Применённые изменения → новый `.md` → все 4 контура работают по обновлённому процессу.

**Важно:** скилл не меняет процесс сам. Он **предлагает**. Это и есть «прозрачная самоэволюция» —
каждое изменение видно, причина видна, апрув человека обязателен.

## Что в MVP (код)

- Логирование сигналов в `signals.log` — минимум **2 источника**: реопены задач + отклонённые
  комменты охраны.
- Анализатор → дайджест + `diff` к `.md`.
- Команда «покажи дайджест» — минимум ручной запуск.

## Как показываем в демо

Финальная сцена (~2:55 → 3:10, растягиваем на 10 сек):
- Подпись на экране: «через 2 недели работы».
- Скилл показывает дайджест: «8 задач вернули из-за отсутствия метрик. Предлагаю добавить пункт в DoR».
- `diff` к `.md`, тимлид жмёт **Apply**.
- Финальный кадр со схемой 5 глаголов.
