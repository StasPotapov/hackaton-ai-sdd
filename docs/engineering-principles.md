# Инженерные принципы оркестратора

> Из сырого груминга. Это не «фичи», а требования к тому, как система работает — они создают
> доверие к агенту и напрямую бьют в критерий «Процесс работы агента». Оркестратор — система
> **под управлением человека**, а не «агент, который просто пишет код».

## 0. Настраиваемый процесс, не захардкоженный пайплайн

Админка-конструктор: из блоков (**этапы, роли, гейты, границы Always / Ask / Never**) собираешь
процесс под любую команду или тип задачи → жмёшь «Применить» → система генерит конфиг Ouroboros
(`SKILL.md` ролей + A2A-оркестрация). В MVP роль конструктора выполняют редактируемые `.md`
(см. [architecture.md](architecture.md)); полноценная админка — Слой 3.

## 1. Не кодить на мутной задаче

Перед кодом агент-аналитик читает тикет и возвращает список проблем: противоречия в требованиях,
недостающие состояния экрана (loading / error / empty / offline), неясные формулировки, скрытые
зависимости. Пока критичные пункты не закрыты человеком — генерация кода не стартует. Поймать на
спеке в разы дешевле, чем переделывать готовый PR.

## 2. Процесс явный, состояние сохраняется

Каждая стадия фиксируется (где задача, что сделано, какой PR) **вне сессии LLM**. Если прогон
упал или систему перезапустили — оркестратор продолжает с того же места, а не «как повезёт со
старым контекстом». **Resume — ответственность системы.**

## 3. Изоляция задачи в песочнице

Каждая задача идёт в своей ветке/worktree: параллельные задачи не мешают друг другу, легко
посмотреть дифф, откатить (чекпойнт `git diff --binary` → при ошибке `reset --hard`) и почистить
за собой. Нужно и для безопасности, и для диагностики.

## 4. «Готово» от агента не считается — проверяем независимо

Слова агента «я закончил» не принимаются на веру: отдельный **quality gate** гоняет тесты, линт
и сборку (плюс опц. LLM-ревью с порогом уверенности). MR открывается только на **зелёной**
проверке, а не по заявлению агента.

## 5. У починки есть лимит

Цикл план → код → проверка → починка ограничен: **максимум попыток, таймаут, бюджет токенов.**
Упёрся в лимит — агент останавливается и зовёт человека, а не чинит одну ошибку часами и не жжёт
бюджет.

## 6. Видно, что агент делает — с первого дня

В пульте по каждой задаче видно: текущая стадия, ветка, PR, результат проверок, потраченная
стоимость, следующее действие. Если человек не понимает, что происходит внутри — он не будет
доверять оркестратору.

## 7. Выборочный контекст и курированная память

Агенту дают ровно нужный кусок — спека, карта модулей (`AGENTS.md`), правила команды — а не весь
репозиторий. И слишком мало, и слишком много контекста одинаково вредят. Память между сессиями
(принятые решения, паттерны) — **курированная**, а не свалка.

## 8. Удобно человеку-оператору

Человек может остановить, поставить на паузу, перезапустить, уточнить задачу и **в любой момент
видеть стоимость.** Оркестратор — система под управлением человека, которой он доверяет.

## Дополнительные способности (из груминга)

- **Поиск дублей** задач в бэклоге.
- **Подробный явный статус задачи** + возврат репортёру незаведённых/недооформленных задач.
- **Захват обсуждения в Slack/СберЧате** → создание задачи в обратку.
- **Сбор инфы перед интервью** при настройке процесса (по Jira-таскам).
