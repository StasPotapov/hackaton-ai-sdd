1. Не кодить на мутной задаче. Перед кодом агент-аналитик читает тикет и возвращает список проблем: противоречия в требованиях, недостающие состояния экрана (loading/error/empty/offline), неясные формулировки, скрытые зависимости. Пока критичные пункты не закрыты человеком — генерация кода не стартует. Поймать на спеке в разы дешевле, чем переделывать готовый PR.

2. Процесс явный, состояние сохраняется. Каждая стадия фиксируется (где задача, что уже сделано, какой PR) вне сессии LLM. Если прогон упал или мы перезапустили систему — оркестратор продолжает с того же места, а не «как повезёт со старым контекстом». Resume — ответственность системы.

3. Изоляция задачи в песочнице. Каждая задача идёт в своей ветке/worktree: параллельные задачи не мешают друг другу, легко посмотреть дифф, откатить (чекпойнт git diff --binary → при ошибке reset --hard) и почистить за собой. Нужно и для безопасности, и для диагностики.
4. «Готово» от агента не считается — проверяем независимо. Слова агента «я закончил» не принимаются на веру: отдельный quality gate гоняет тесты, линт и сборку (плюс опционально LLM-ревью с порогом уверенности). MR открывается только на зелёной проверке, а не по заявлению агента.
5. У починки есть лимит . Цикл план→код→проверка→починка ограничен: максимум попыток, таймаут, бюджет токенов. Упёрся в лимит — агент останавливается и зовёт человека, а не чинит одну и ту же ошибку часами и не жжёт бюджет.
6. Видно, что агент делает — с первого дня. В пульте по каждой задаче видно: текущая стадия, ветка, PR, результат проверок, потраченная стоимость, следующее действие. Если человек не понимает, что происходит внутри — он не будет доверять оркестратору.
7. Выборочный контекст и память, а не огромный промпт. Агенту дают ровно нужный кусок — спека, карта модулей (AGENTS.md), правила команды — а не весь репозиторий. И слишком мало, и слишком много контекста одинаково вредят. Память между сессиями (принятые решения, паттерны) — курированная, а не свалка всего подряд.
8. Удобно человеку-оператору. Человек может остановить, поставить на паузу, перезапустить, уточнить задачу и в любой момент видеть стоимость. Оркестратор — это система под управлением человека, которой он доверяет, а не только «агент, который пишет код».

# Хакатон: скилл «Definition of Ready» для SDD-процесса

> Рабочее имя: **Ready** (варианты: «Готовность», «Покажи пять»)
> Платформа: скилл для нашего аналога Claude Code (GigaCode), поверх MCP Atlassian (Jira + Confluence)

---

## Суть в одном предложении

**«Покажи мне 5 своих задач — и я покажу тебе твой процесс. А потом проведу любую задачу до готовности, пока к ней не останется вопросов.»**

Это не генератор документов. Это **гейт готовности (Definition of Ready) как скилл**: сырая Jira-задача входит → выходит чёткое ТЗ, готовое к передаче другой команде, без единого «а что ты имел в виду?».

---

## Зачем (настоящая боль)

В компании все хотят SDD, но никто не может описать, как он выглядит. Поэтому:
- регламенты на 40 страниц никто не читает,
- внедрение буксует на «давайте сначала договоримся о процессе».

Идея бьёт ровно сюда: **процесс не описываем — добываем из примеров.** Порог входа стремится к нулю.

---

## Как это устроено (движок из 4 шагов)

1. **Калибровка** (один раз на команду).
   Скилл смотрит 5 эталонных задач из вашей Jira → выводит ваш *неявный* стандарт: чек-лист готовности + шаблоны (ТЗ / PRD / аналитика) → **пишет их в свои markdown-файлы**.
   *Команда может править эти файлы — это и есть вся «настройка». Никакого кода и UI.*

2. **Захват.**
   Скилл тянет задачу из Jira через MCP Atlassian, подтягивает контекст: связанные задачи, эпик, похожие прошлые.

3. **Петля готовности** (ров проекта).
   Пишет черновик ТЗ → **сам себя критикует** по чек-листу → задаёт человеку уточняющие вопросы, если данных не хватает → переписывает → повторяет, **пока замечаний не ноль** (readiness score → порог).

4. **Выгрузка.**
   Кладёт готовое ТЗ/PRD в Confluence, дозаполняет Jira (описание, поля), ставит метку **Ready**.
   *Для разработчика — опционально черновик PR (стретч, не звезда демо).*

**Human-in-the-loop:** на каждом стыке — «дальше / поправить / стоп». Это не чёрный ящик, а конвейер с чекпоинтами.

---

## Один движок — все роли

Меняется только чек-лист «что значит готово»:

| Роль | Вход | Выход (готово, когда…) |
|---|---|---|
| Аналитик | сырая гипотеза | ТЗ передаётся в разработку без вопросов |
| Продакт | идея фичи | PRD согласован, метрики и scope зафиксированы |
| Дизайнер | задача на макет | бриф + критерии, готовые к проектированию |
| Разработчик | готовое ТЗ | (стретч) черновик PR поверх него |

---

## Демо (≈3 минуты — то, что выигрывает)

- **0:00 — Проблема.** Одно предложение: «Все хотят SDD. Никто не знает как. Внедрение = ноль.»
- **0:20 — Самонастройка вживую.** Скармливаем 5 реальных задач компании-жюри → скилл выдаёт **черновик ИХ стандарта**. *«Мы не принесли вам процесс. Сейчас найдём ваш.»*
- **1:00 — Превращение.** Берём мусорную задачу («пофиксить оплату @вася, горит») → **сплит-экран** → петля крутится → **readiness score 25% → 95%** → готовое ТЗ + ссылка в Confluence. Вживую.
- **2:00 — Вторая роль.** Тот же скилл прогоняет задачу аналитика, другой чек-лист. *«Для всей компании, не только для разработки.»*
- **2:30 — Финал.** «А поверх готового ТЗ для дева — черновик PR. Но звезда — вот это превращение.»

**Страховка (Грумбал):** сид данных заранее, записанный фолбэк в кармане. Магия настоящая, но прогон отрепетирован — никакого «всё вживую без сетки».

---

## Почему выигрывает (чего не было у «просто генератора»)

| Слабое место | Чем закрыли |
|---|---|
| «Это же просто ChatGPT пишет PRD» | **Ров:** обучение на вашем бэклоге + замкнутая петля самокритики. Генерацию покажут все — это никто |
| «Дайте гибкую базу» = пустота | База — это **мнение**: жёсткий формат, мягкое содержание. Спорить с конкретным = находить свой процесс |
| «Процесс» (скучно жюри) | **Превращение на сплит-экране** + ползущая метрика 25%→95% |
| «Для разработчиков» | Один примитив, две роли на сцене — продаёт менеджменту |
| Высокий порог внедрения | **Самонастройка вживую** из 5 задач → ноль внедрения |

## Что выкинуть ради победы

Авто-PR с рабочим кодом — риск без награды на сцене. Оставить одной фразой как «дальше доведём до кода».

---

## Roadmap: слои фич

Принцип питча (Грумбал/Виктор): **маленькое и железное — показываем, большое и неизбежное — рассказываем.**
Связка одной фразой: *«Сегодня он доводит задачу до готовности. Завтра — держит нить от идеи до релиза.»*

### Слой 1 — успеем и вау (в демо к субботе)

| Фича | Почему вау | Почему дёшево |
|---|---|---|
| **Readiness score** 25%→95% | менеджер покупает растущую полоску | уже ядро петли |
| **Три вопроса сеньора** — скилл задаёт ровно 3 самых больных уточнения, не 20 | выглядит как мышление сеньора, не как форма | один промпт поверх петли |
| **«Это дубль!»** — находит похожую/дублирующую задачу в бэклоге | момент «откуда он знает?» | поиск по Jira через MCP, уже есть |
| **DoR-линтер на стенде** — вставь любую задачу → «готовность 40%, нет критериев приёмки» | прохожий становится фанатом за 5 сек, продаёт без вас | срез петли без полного пайплайна |
| **Diff-коммент в Jira** — «что я добавил и почему» | прозрачность → доверие к агенту | один коммент через MCP |

### Слой 2 — стретч (если останется время)

- Черновик PR поверх готового ТЗ (для дева).
- **Оценка по аналогии** — story points по похожим прошлым задачам (тоже добывается из бэклога — усиливает ров).
- Нотификация (Slack/почта), когда задача стала **Ready**.

### Слой 3 — будущее (рассказываем со сцены как мечту)

- **Сквозная нить (оркестратор):** идея → PRD → ТЗ → задачи → PR → релиз одним графом. Отвечает на «почему эта строчка кода существует?» за секунду.
- **Импакт-анализ:** «меняем это требование — вот какие задачи/код/тесты задеты». Две недели совещаний → один запрос.
- **Живой стандарт:** новое замечание на ревью, которого не было в чек-листе → скилл предлагает дописать правило. Регламент **затачивается сам, из практики**.
- **Оркестрация ролей:** задача автоматически идёт по эстафете аналитик → продакт → дизайнер → dev, каждому — готовый вход.
- **Метрики процесса:** дашборд — где задачи застревают, среднее время до Ready, узкие места.
- **Онбординг новичка:** «как у нас принято писать ТЗ?» — отвечает на ваших же примерах.
