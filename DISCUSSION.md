# Синк с командой: решения на «ок»

> Проект формализован одним человеком — этот файл нужен, чтобы **каждое важное решение окнула
> вся команда**. Прямо в файле: ставим статус, пишем комментарии, фиксируем изменения.
> Заявка — [PROPOSAL.md](PROPOSAL.md), детали — в [docs/](docs/README.md).

**Как работаем со статусами:**

- `[ ]` — не обсуждали
- `[x] ОК` — согласны, едем
- `[?] Обсудить` — есть вопросы/сомнения (написать какие)
- `[!] Менять` — не согласны (написать, что предлагаем вместо)

### 1.2. Продаём полный конвейер идея → PR, реализуем поэтапно

Заявка продаёт **оркестратор SDD-конвейера целиком** (сигнал → ТЗ → аналитика → дизайн → код →
PR). Прототип **гарантирует** этапы до готового ТЗ + охрану + эволюцию; этап кода — ранняя
версия (один happy-path прогон до PR). Конвейер останавливается на любом этапе без потери
ценности — это позиционируется как фича (управляемая деградация), а не как срезанный скоуп.

- Статус: `[ ]`
- Комментарии:

### 1.3. Звезда демо — превращение сырой задачи в готовое ТЗ

Показываем железное; этап кода — только отрепетированный happy-path (или одной фразой,
если не дозреет). Продаём конвейер, показываем готовность.

- Статус: `[ ]`
- Комментарии:

### 1.4. Конструктор процесса из блоков — за пределами MVP

Настройка процесса в MVP — правка `.md`-файлов (это наш тезис «один источник правды»).
UI-конструктор (этапы/роли/гейты из блоков → конфиг Ouroboros) — целевое видение, в заявке
упоминается как будущее, не обещается.

- Статус: `[ ]`
- Комментарии:

### 1.5. Каналы связи — агностичны; для демо Jira + Telegram

Вход и выход не привязаны к конкретному каналу: **сигнал — из любого подключённого канала**
(тег в чате, встреча, Jira, хоть командная строка), **обратная связь — в любой канал, куда
Ouroboros умеет писать** (MCP/транспорты). Дефолт для аудируемости: вопросы и замечания —
**комментарием в самой Jira-задаче** (история не теряется). **Для показа используем Jira +
Telegram**: в Telegram можно тегать агента, и агент пишет туда же (telegram-bridge — штатный
транспорт Ouroboros).

## 2. Три главные фичи (звёзды прототипа — окнуть каждую)

### 2.1. Заводит задачи сам, по вашему процессу

Тег агента в чате или встреча из Jazz → суть вытянута → готовая задача в Jira по стандарту
команды. «Обсуждение больше не теряется».

- Статус: `[ ]`
- Комментарии:

### 2.2. Охраняет входящие

Фоновый мониторинг очереди Jira (cron-механизм Ouroboros) → сверка новой задачи с процессом →
авто-коммент автору «чего не хватает» (нет критериев приёмки, макета, метрики).

- Статус: `[ ]`
- Комментарии:

### 2.3. Самоэволюция процесса

Раз в спринт анализ задач команды (что влетало, менялось, реопенилось) → дайджест + `diff` к
`.md` → тимлид жмёт Apply/Reject. Агент не меняет процесс сам — только предлагает.
Это главный дифференциатор для жюри (требование хакатона о самосовершенствовании).

- Статус: `[ ]`
- Комментарии:

### Новые фичи — предложены 02.07, обсудить и решить, куда кладём (ядро / усилители / видение)

### 2.4. Доска процесса (веб-панель + дашборд)

Панель, которая **показывает процесс команды вживую**: где каждая задача (этап конвейера),
статус гейтов, следующее действие. **Своё не пишем** — используем готовое:
у Ouroboros уже есть web-UI с Dashboard (Logs / Evolution / Costs / Activity), а наш скилл может
объявить **свою вкладку** (`ui_tab`, host-rendered виджеты: таблица/карточки) — панель процесса
без собственного фронтенда. Плюс у команды остаётся обычная Jira-доска (статусы и метки Ready
скилл проставляет сам). Обсудить: достаточно ли вкладки Ouroboros + Jira-доски, или ищем внешнее
решение.

- Статус: `[?] Обсудить`
- Куда кладём (ядро / усилитель / видение): ___
- Комментарии:

### 2.5. Контроль стоимости задачи (токены → деньги)

На той же панели — **сколько токенов/денег потрачено на каждую задачу и этап**: видимая
стоимость прогона, бюджет-лимит на задачу («этап кода не дороже X»), стоп при превышении.
Ouroboros считает токены и цену из коробки (Dashboard → Costs, per-task cost threshold) — фича
в том, чтобы **вынести стоимость к задаче на нашу панель** и сделать её аргументом доверия:
«вы всегда видите, во что обходится агент». Хорошо продаётся жюри (инженерный инвариант
«видимая стоимость» становится продуктовой фичей) и подкрепляет финмодель фактами с первого дня.
