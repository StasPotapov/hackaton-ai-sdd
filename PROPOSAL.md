## 1. Описание кейса (вес 30%)

### Зачем (проблема)

В компании все хотят SDD (Spec-Driven Development), но никто не может описать, **как он выглядит**
именно у их команды. Следствия:

- регламенты на 40 страниц никто не читает;
- внедрение процесса буксует на этапе «давайте сначала договоримся, как мы работаем»;
- сырые задачи уходят в разработку → возвраты, переделки, бесконечные «а что ты имел в виду?»;
- готовые код-агенты стартуют на мутных задачах и выдают правдоподобный, но неверный код,
  который дольше проверять, чем писать заново;
- знание процесса живёт в головах сеньоров и теряется при онбординге.

### Что (суть инициативы)

**Самоэволюционирующийся SDD-оркестратор**: ловит сигнал из любого канала (задача в Jira, тег
агента в чате, встреча в **Jazz**), проводит его через SDD-процесс *вашей* команды по этапам —
**готовность (ТЗ) → аналитика → дизайн → код → PR** — с гейтами и human-in-the-loop на каждом
стыке. Каждый этап — самостоятельная ценность: конвейер можно остановить на готовом ТЗ, этап
кода включается тогда, когда команда ему доверяет.

Ключевая идея: **процесс не описываем — добываем из примеров.** Порог входа стремится к нулю.

Формула продукта — пять глаголов, замкнутый цикл:

> **Добывает → Применяет → Охраняет → Учит → Эволюционирует.**

Один источник правды — редактируемые `.md`-файлы (чек-лист готовности + шаблоны + описание
этапов). Один движок работает для всех ролей — меняется только чек-лист «что значит готово»:

| Роль | Вход | Выход (готово, когда…) |
|---|---|---|
| Аналитик | сырая гипотеза | ТЗ передаётся в разработку без вопросов |
| Продакт | идея фичи | PRD согласован, метрики и scope зафиксированы |
| Дизайнер | задача на макет | бриф + критерии для проектирования |
| Разработчик | готовое ТЗ | **PR на зелёном quality gate** |

### Как (в двух словах)

1. **Калибровка** — скилл смотрит 5 эталонных задач команды → выводит её неявный стандарт →
   пишет чек-лист, шаблоны и этапы конвейера в `.md`.
2. **Конвейер** — сигнал → задача → петля готовности (черновик → самокритика по чек-листу →
   уточняющие вопросы человеку) → готовое ТЗ → следующие этапы по процессу команды, вплоть до
   кода в изолированной ветке и PR на зелёном quality gate.
3. **Выгрузка** — артефакты этапов в Confluence/Jira, метка **Ready**, PR в репозитории.

### Отстройка (почему это не «просто ChatGPT пишет PRD» и не ещё один код-агент)

- **Ров:** обучение на **вашем** бэклоге + замкнутая петля самокритики (не разовая генерация).
- **Не код-агент, а управляемый конвейер:** код не стартует на мутной спеке (интервью-гейт),
  «готово» агента проверяется независимым quality gate, у каждого действия есть апрув и откат.
- **База — это мнение:** жёсткий формат, мягкое содержание. Команда спорит с конкретным
  чек-листом → в споре находит свой процесс. «Гибкая база» = пустота, мы даём позицию.
- **Самоэволюция процесса, а не модели:** эволюционирует описание процесса (`.md`) через `diff`
  и апрув человека — для банка это не риск, а усиление контроля.

## 2. Эффект от внедрения (вес 20%)

**Продаём не скорость, а качество и предсказуемость**: меньше переделок, единый процесс,
аудируемость. Эти эффекты **не зависят от скорости инференса** — сохраняются и на моделях
внутреннего контура (GigaChat / Cloud.ru), деньги — следствие качества.

### За счёт чего достигается эффект

1. **Меньше переделок спеки** — дефект ловится до разработки, а не в готовом PR. Правило
   **1:10:100**: дефект на спеке на порядок дешевле дефекта в коде и на два порядка дешевле
   прод-инцидента.
2. **Меньше переделок кода** — этап кода не стартует на мутной задаче (clarify-гейт); PR
   открывается только на зелёном quality gate (обязательные тесты/линт/сборка).
3. **Меньше времени на груминг** — рутину уточнений берёт скилл, сеньор подключается точечно.
4. **Быстрее онбординг** — новичок спрашивает «как у нас принято?» и получает ответ на примерах.
5. **Ноль стоимости внедрения процесса** — не пишем регламент, добываем из 5 задач.
6. **Качество растёт со временем** — самоэволюция под ретро-гейт режет пул повторяющихся
   переделок от спринта к спринту (отложенный множитель эффекта).

### Бизнес-метрики AS IS → TO BE

| Метрика | AS IS (допущение) | TO BE (цель) |
|---|---|---|
| Среднее время задачи до статуса Ready | 2–4 дня (растянутый груминг) | часы (один прогон + апрув) |
| Доля возвращённых задач (Reopened / back to Backlog) | ~20–30% | < 10% |
| Кол-во уточняющих итераций «а что имел в виду?» на задачу | 3–5 | 0–1 |
| Итераций до зелёного MR (на этапе кода) | 2–4 | 1–2 |
| Время онбординга новичка до первой самостоятельной задачи | недели | дни |
| Стоимость внедрения SDD-процесса | месяцы согласований + регламент | ~0 (5 задач → стандарт) |

> ⚠️ Значения AS IS — **допущения**; заменяются замером по плану
> [docs/asis-measurement-plan.md](docs/asis-measurement-plan.md) (выгрузка Jira команды-пилота +
> выборочный ручной подсчёт, ~1–2 часа работы).

### Модель эффекта (Value to Cost)

Формула экономии (годовая, на одну команду):

```
Экономия = (Δчасов_на_задачу × число_задач_в_год × ставка_час)
         + (Δчасов_переделок × число_возвратов_в_год × ставка_час)
         + (Δчасов_онбординга × число_новичков_в_год × ставка_час)
```

**Расчёт на модельную команду ~10 человек, ~300 задач/год (все значения — допущения,
помечены ⚠️ и заменяются фактами):**

| Статья дохода (экономии) | Расчёт | Эффект/год |
|---|---|---|
| Сокращение груминга | 3 ч/задача × 300 задач × 3 000 ₽/ч ⚠️ | 2,70 млн ₽ |
| Сокращение переделок | 8 ч × 60 возвратов × 3 000 ₽/ч ⚠️ | 1,44 млн ₽ |
| Ускорение онбординга | 40 ч × 5 новичков × 3 000 ₽/ч ⚠️ | 0,60 млн ₽ |
| **Итого экономия** | | **≈ 4,7 млн ₽** |

Ставка часа — **полная стоимость** сотрудника (fully loaded): месячная ЗП × ~1,5 (налоги,
накладные) / 168 ч. При средней ЗП IT-специалиста ~340 тыс ₽/мес → ~3 000 ₽/ч ⚠️.

| Статья расхода | Расчёт | Затраты/год |
|---|---|---|
| Токены LLM: этапы до ТЗ | ~150K вх. + 15K вых. токенов/задача ≈ 0,7 $ ≈ 100 ₽ ⚠️ × 300 задач | 0,03 млн ₽ |
| Токены LLM: этап кода (цикл план→код→проверка с лимитом) | ~1 200 ₽/задача ⚠️ × 150 задач | 0,18 млн ₽ |
| Инфраструктура | Ouroboros — внутренняя платформа Банка; MCP-сервер Atlassian self-hosted | ~0,05 млн ₽ ⚠️ |
| Поддержка/сопровождение | 0,2 FTE × полная стоимость года (~5,4 млн ₽) ⚠️ | 1,08 млн ₽ |
| **Итого затраты** | | **≈ 1,35 млн ₽** |

> Оценка токенов: петля готовности — несколько LLM-проходов (черновик, самокритика, вопросы,
> переписывание) по ~150K входных / ~15K выходных токенов на задачу; по ценам моделей среднего
> класса (~3 $/15 $ за 1M токенов через OpenRouter) это ≈ 0,5–1 $ на задачу. Этап кода с циклами
> починки — ≈ 5–15 $ на задачу, взято 1 200 ₽ с запасом. Ouroboros ведёт учёт стоимости токенов
> из коробки (Dashboard → Costs) — фактическая цена прогона видна с первого дня прототипа.

**Value to Cost ≈ 4,7 / 1,35 ≈ 3,5×** на одну команду (при допущениях выше).
При тиражировании эффект мультиплицируется: скилл один, затраты поддержки растут слабо →
на 5 команд V2C ≈ 10×+.

### Нефинансовые эффекты

- **AI-adoption — стратегический KPI Банка:** каждая задача через конвейер — управляемое и
  аудируемое применение AI в разработке (растёт доля AI-разработки под контролем, а не «теневое»
  использование ассистентов).
- **Аудируемость под банковскую регуляторику:** стоп-гейты **Always / Ask / Never** + логи всех
  действий + апрув человека — у каждого изменения есть diff, причина, апрувер и откат.
- **Предсказуемость поставки:** единый процесс для всех задач, меньше дефектов доезжает до прод
  (обязательные проверки в quality gate).

### Тиражируемость эффекта

Эффект мультиплицируется: один скилл → любое число команд без доработки (см. критерий 6).

## 3. Процесс работы агента (вес 15%)

### Пошаговый сценарий

**Шаг 0. Калибровка через интервью (один раз на команду).**
Скилл не с нуля — в него заложена база «каким должен быть хороший процесс» (обязательные атрибуты
задачи, этапы конвейера). Тимлид указывает 5 эталонных задач → скилл накладывает их на базу и,
где данных мало, **доспрашивает команду** (короткое интервью) — так вытягивает ваш процесс →
пишет чек-лист, шаблоны и этапы в `checklist.md` / `template.md`. Команда правит `.md` — это вся
настройка, без кода.

**Шаг 1. Захват сигнала.**
Скилл заводит задачу сам — из чата или из встречи в **Jazz** (по тегу агента вытягивает суть в
готовую задачу по процессу) — либо тянет существующую задачу из Jira через MCP Atlassian +
контекст: связанные задачи, эпик, похожие прошлые. Проверяет на дубли.

**Шаг 2. Этап готовности (первый гейт конвейера).**
```
draft ТЗ по template.md
   → самокритика по checklist.md → чего не хватает
   → уточняющие вопросы человеку (human-in-the-loop)
   → переписать с учётом ответов → готовое ТЗ
```
Пока критичные вопросы спеки не закрыты человеком — следующие этапы не стартуют.

**Шаг 3. Выгрузка и следующие этапы.**
Готовое ТЗ/PRD → Confluence; дозаполнение полей Jira; метка **Ready**. Дальше задача идёт по
этапам процесса команды: аналитика → дизайн → **код** (изолированная ветка/worktree, независимый
quality gate: тесты + линт + сборка) → **PR открывается только на зелёной проверке**. Конвейер
останавливается на любом этапе — готовое ТЗ уже самостоятельная ценность.

**Шаг 4. Охрана и эволюция (фоново, по расписанию).**
Мониторит очередь Jira: в заведённой задаче сверяет с процессом и, если чего-то не хватает,
**пишет комментарий в самой задаче и пингует автора** в подключённом канале (для демо —
Telegram). Копит сигналы (реопены, доправки)
→ раз в спринт смотрит на задачи команды (что влетало, как менялось, реопены) и предлагает
`diff` к `.md` → тимлид апрувит.

### Как встраивается в текущий процесс

- Поверх **существующих** Jira, Confluence и репозитория — новые инструменты команде не нужны.
- **Каналы — агностичны:** сигнал на вход из любого подключённого канала (тег в чате, встреча,
  Jira, командная строка), обратная связь — в любой канал, куда Ouroboros умеет писать
  (MCP/транспорты). Дефолт для аудируемости — **комментарий в Jira-задаче** (история в задаче)
  + пинг автора в чате со ссылкой. В прототипе показываем Jira + Telegram.
- **Human-in-the-loop** на каждом стыке этапов: «дальше / поправить / стоп». Не чёрный ящик.
- Скилл **никогда не меняет процесс сам** — только предлагает, апрув человека обязателен.
- Инженерные гарантии доверия: явное состояние вне сессии LLM (resume), изоляция задач в
  ветке/worktree, независимый quality gate, лимиты на попытки/токены, видимая стоимость,
  курированная память.

## 4. Функции прототипа / MVP (вес 15%)

Прототип — **конвейер с явными этапами**. К демо гарантируем этапы «сигнал → готовность →
выгрузка» плюс охрану и эволюцию; этап кода включён в прототип как ранняя версия (один
happy-path прогон до PR). Конвейер спроектирован так, что этап кода **скипается в любой момент
без потери ценности** — это управляемая деградация, а не срезание скоупа.

### Три главные фичи (звёзды прототипа)

1. **Заводит задачи сам, по вашему процессу** — тег агента в чате или встреча из **Jazz** →
   готовая задача в Jira по стандарту команды.
2. **Охраняет входящие** — мониторинг очереди Jira по расписанию → сверка с процессом →
   авто-коммент автору о том, чего не хватает.
3. **Самоэволюция процесса** — раз в спринт анализ задач команды → предложение улучшить процесс
   (`diff` к `.md` → апрув человека).

### Ядро MVP (фундамент под фичи)

1. **Калибровка с интервью** — база «хорошего процесса» + 5 задач + доспрос → `.md`.
2. **Этап готовности** — самокритика по чек-листу + уточняющие вопросы.
3. **Выгрузка** в Confluence + метка `Ready`.
4. **Онбординг Q&A** — ответы на «как у нас принято?» по примерам.
5. **Петля эволюции (минимум):** `signals.log` (≥2 источника) + дайджест + `diff` к `.md`.
6. **Этап кода (ранняя версия):** готовое ТЗ → изолированная ветка → код → quality gate →
   черновик PR. Один отработанный happy-path прогон.

### Слой 1 — усилители (если успеем)

| Фича | Ценность |
|---|---|
| **Три вопроса сеньора** | ровно 3 главных уточнения, не форма из 20 полей |
| **«Это дубль!»** | находит похожую/дублирующую задачу в бэклоге |
| **DoR-линтер** | вставь задачу → «готовность 40%, нет критериев приёмки» |
| **Readiness score** | наглядный рост готовности задачи |
| **Доска процесса** | вкладка в web-UI Ouroboros (`ui_tab`): где каждая задача, этап, гейты, следующее действие — без собственного фронтенда |
| **Стоимость задачи** | токены → деньги per задача/этап на той же панели + бюджет-лимит (учёт стоимости — штатный в Ouroboros); видимая стоимость как аргумент доверия |

### Целевое видение (рассказываем, не обещаем в MVP)

Конструктор процесса из блоков (этапы/роли/гейты/границы Always·Ask·Never → генерит конфиг
Ouroboros), сквозная нить идея→релиз одним графом, импакт-анализ требований, дашборд метрик
процесса. См. [docs/orchestrator.md](docs/orchestrator.md).

## 5. Ограничения (вес 10%)

### Источники данных

- **Jira** — задачи, поля, статусы, комментарии, связи (через MCP Atlassian).
- **Confluence** — выгрузка артефактов, метки.
- **Репозиторий (GitLab/аналог)** — ветки, PR, результаты проверок (этап кода).
- **Каналы связи** (Telegram / Slack / СберЧат / …) — вход сигналов и обратная связь; каналы
  агностичны, подключается любой доступный Ouroboros транспорт. В прототипе — Telegram.
- Данные — **внутренний контур команды**; скилл не требует внешних датасетов.

### Кибербезопасность

- **Ouroboros — внутренняя платформа Банка** и нативно поддерживает модели внутреннего/российского
  контура (GigaChat, Cloud.ru Foundation Models) → в проме чувствительные данные могут не
  покидать контур. Внешние LLM (OpenRouter) — только на хакатоне, с контролем передаваемых
  данных и маскированием/исключением чувствительных полей.
- Изоляция задач в песочнице (ветка/worktree), чекпойнты и откат.
- **Границы автономии Always / Ask / Never:** чтение — свободно; запись / exec / push — через
  Approval Gate; разрушительные операции — заблокированы на уровне платформы (промпт-инъекция
  границу не обходит).
- **Апрув человека** на все изменения → нет автономных действий с необратимыми последствиями;
  PR — только через независимый quality gate.
- Доступы через сервисную учётку с минимально необходимыми правами (least privilege);
  MCP-инструменты в Ouroboros проходят per-call safety-check, токены маскируются.

### Регуляторные ограничения

- Банковская тайна и ПДн в задачах → обработка в рамках внутренних политик Банка; при
  необходимости — обезличивание (в т.ч. расшифровок встреч перед LLM).
- Аудируемость: обратная связь — комментарии в Jira (история в задаче), все предложения скилла
  логируются и проходят ревью человека → трассируемость решений.

> ⚠️ Точные формулировки регуляторных требований согласовать с ИБ/комплаенс подразделения перед
> продуктивом. Здесь — рамка, не юридическое заключение.

## 6. Реализуемость (вес 10%)

### Пошаговый процесс реализации прототипа на Ouroboros

Всё перечисленное — **штатные механизмы платформы Ouroboros**, наш код — конфигурация, промпты
и тонкий plugin-слой:

1. **Скилл** — тип `extension` (манифест `SKILL.md` + `plugin.py` через PluginAPI); процесс
   команды — в `.md`-файлах скилла.
2. **MCP Atlassian** — через встроенный MCP-клиент Ouroboros (настройка `MCP_SERVERS`,
   транспорт streamable HTTP/SSE): чтение задач, поиск, комментарии, выгрузка в Confluence.
3. **Фоновый мониторинг очереди** (фича «охраняет») — блок `scheduled_tasks` (cron) в манифесте
   скилла; планировщик Ouroboros сам будит агента по расписанию. Тот же механизм — для
   спринтового дайджеста эволюции.
4. **Канал связи для демо** — telegram-bridge, штатный transport-скилл Ouroboros (тег агента в
   чате как сигнал, ответы/пинги туда же); архитектура канал-агностична — подключается любой
   транспорт/MCP.
5. **Роли-агенты конвейера** — субагенты Ouroboros: model lanes (main/heavy/light/review) под
   вес этапа, изоляция каждого acting-субагента в **git-worktree** и интеграция патча с проверкой
   — из коробки.
6. **Петля готовности + калибровка** — промпты ролей + `checklist.md`/`template.md`.
7. **Наблюдаемость и стоимость** — встроенный Dashboard Ouroboros (Logs / Costs / Activity),
   учёт цены токенов на задачу.
8. Подготовка демо-данных (сид) и записанный фолбэк.

### Масштабируемость (в рамках подразделения)

Единый движок, вся специфика — в `.md`. Новая команда = свои 5 задач → свой стандарт. Никакого
кода под каждую команду.

### Переиспользование (другими подразделениями)

Механика «покажи 5 задач → получи свой процесс» работает для любого подразделения (аналитика,
продакт, дизайн, разработка) без изменения кода — меняется только чек-лист и набор этапов.

### Риски и нивелирование

| Риск | Нивелирование |
|---|---|
| Качество стандарта, добытого из 5 задач | Human-in-the-loop: тимлид правит `.md`; петля эволюции дозатачивает из практики |
| Этап кода не успеет «дозреть» к демо | Конвейер деградирует до готового ТЗ без потери ценности — этапы независимы |
| Чувствительные данные в задачах | Внутренний контур (GigaChat/Cloud.ru в проме), обезличивание, апрув человека, least privilege |
| Недоверие команды к агенту | Прозрачный diff-коммент, видимая стоимость, «стоп» на каждом шаге |
| Галлюцинации / неверные уточнения | Порог readiness score, лимит итераций, независимый quality gate, финальный апрув |
| Сбой на демо | Сид данных заранее + записанный фолбэк |
